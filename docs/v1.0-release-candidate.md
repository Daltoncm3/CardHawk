# CardHawk Scouting Engine v1.0 Release Candidate

## RC Version

`v1.0.0-rc.1`

## Release Scope

This Release Candidate covers the CardHawk Scouting Engine only.

Included in scope:

- eBay marketplace scouting as the only active live marketplace.
- Marketplace abstraction with inactive mock marketplace support.
- Scouting, comp, sold sales, market value, ROI, risk, confidence, quality, grading, trend, population, market intelligence, history, learning, sales velocity, decision, validation, prediction accuracy, notification, system health, and engine metrics capabilities.
- Persistent engine state using atomic JSON writes where implemented.
- Read-only metrics and health visibility.
- Paper/production mode visibility.
- Operator audit logging for approved manual operational actions.
- Backup, restore, release smoke, and architecture freeze documentation.

Out of scope for v1.0:

- Portfolio Engine.
- Bankroll Management Engine.
- Sell/Exit Strategy Engine.
- SaaS account system.
- Live second marketplace integration.
- Storage-key migration.
- Marketplace-qualified storage migration.
- Automated purchase execution.

## Completed Architecture Phases

- Phase 7.1: Prediction accuracy persistence, history atomic writes, decision validation persistence.
- Phase 7.2: Marketplace abstraction foundation, eBay adapter, marketplace registry, inactive mock marketplace.
- Phase 7.3: Read-only listing identity helpers and API lookup compatibility.
- Phase 7.4: Read-only engine metrics and metrics dashboard.
- Phase 7.5: Atomic main store persistence, route-method hardening, app store extraction, dependency-free smoke tests.
- Phase 7.6: Scout scanner orchestration extraction and notification idempotency persistence.
- Phase 7.7: Startup config readiness, operator audit logging, paper mode guardrails, backup/restore documentation, release smoke checklist, architecture freeze documentation.

## Required Automated Tests

Before tagging `v1.0.0-rc.1`, run:

- `npm test`
- `npm run test:smoke`

Required result:

- All tests pass.
- Smoke tests do not touch production `data/` files.
- No unexpected test flakes.
- No runtime source changes are made as part of test execution.

Before tagging `v1.0.0-rc.1`, complete the manual checks in `docs/release-smoke-checklist.md`.

Minimum manual verification:

- `/health` loads.
- `/api/status` returns valid JSON.
- `/api/metrics` returns valid JSON.
- `/metrics` renders.
- Main dashboard renders.
- Alerts page renders.
- Rejections page renders.

## Release Blockers Checklist

Block `v1.0.0-rc.1` if any item below is true:

- `npm test` fails.
- `npm run test:smoke` fails.
- App fails to start.
- `/health` fails.
- `/api/status` fails.
- `/api/metrics` fails.
- Railway deployment fails or remains unhealthy.
- Required production environment variables are missing unexpectedly.
- Config readiness reports unexpected critical issues.
- Runtime logs expose secrets, passwords, auth headers, API keys, or tokens.
- Persistent state cannot be loaded and no safe backup or recovery path exists.
- Unexpected state corruption warnings appear during release validation.
- Notification idempotency does not survive restart.
- Side-effect GET routes are reintroduced.
- eBay marketplace failures are not surfaced safely.
- Scoring changed without explicit release-blocker approval.
- Deal Gate changed without explicit release-blocker approval.
- BUY_NOW behavior or internal naming changed without explicit release-blocker approval.
- Alert or notification behavior changed without explicit release-blocker approval.
- Scan timing changed without explicit release-blocker approval.
- Storage keys or persisted state formats changed without explicit release-blocker approval.
- No current backup exists before production deployment.
- Rollback path is unknown.

## Tagging Process

Use this process after automated tests, manual verification, and sign-off are complete:

1. Confirm the worktree is clean.
2. Confirm the final commit hash.
3. Confirm Railway has deployed the intended commit successfully.
4. Create the release candidate tag:

   ```sh
   git tag v1.0.0-rc.1
   ```

5. Push the tag:

   ```sh
   git push origin v1.0.0-rc.1
   ```

6. Record the tag, commit hash, Railway deployment status, backup timestamp, and operator sign-off in this document or the release record.

## Deployment Verification

After deployment:

- Confirm Railway build completed successfully.
- Confirm Railway service is healthy.
- Confirm deployed commit matches the RC commit.
- Confirm `/health` loads.
- Confirm `/api/status` shows the expected `cardhawkMode`.
- Confirm `/api/status` shows expected config readiness.

## Rollback Procedure

If rollback is required:

1. Stop the release process.
2. Identify the previous known-good Git commit.
3. Identify the previous known-good Railway deployment.
4. Confirm whether runtime data must also be restored.
5. If data restore is required, follow `docs/backup-and-restore.md`.
6. Prefer restoring the full `data/` directory from one timestamp instead of partial file restore.
7. Confirm notification state implications before restoring old backups.
8. Confirm operator audit implications before restoring old backups.
9. Redeploy the previous known-good commit or use the previous known-good Railway deployment.
10. Verify `/health`, `/api/status`, `/api/metrics`, and `/metrics`.
11. Record the rollback reason, timestamp, operator, restored commit, and restored backup if applicable.
